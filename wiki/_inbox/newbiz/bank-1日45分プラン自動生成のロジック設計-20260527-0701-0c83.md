# Testall 1日45分プラン自動生成のロジック設計

## 1. 設計の目的と前提

**目的**: `[[entities/testall]]` のコア機能「テスト結果→弱点特定→**今日の45分タスクに自動変換**」のAIロジックを具体化する。
**前提**:
- ターゲットは「塾なし独学受験生」。彼らは「何を勉強すべきか」で迷い、挫折する。
- 大井の強みはFounder-Market Fit（大井自身が受験世代）。この直感をコード化する。
- **証拠**: 既存のStudyplusやatama+は「計画の立て方」や「問題集の提示」までだが、「次の45分」の具体的な行動まで落とし込む機能は弱い。
- **明日使う1人**: 明日テスト返却をもらい、復習に困っている高校2年生（大井の友人層）。

## 2. 入力データの構造（What goes in）

AIが判断するための入力データは、以下の3層で構成する。

### 2.1. 静的データ（ユーザープロファイル）
- **志望校**: 難易度（MARCH/関関同立/地方国公立等）
- **得意科目/不得意科目**: 自己申告
- **残り日数**: 受験日からのカウント（例：残り180日）
- **学習スタイル**: 朝型/夜型/集中時間帯

### 2.2. 動的データ（テスト結果）
- **科目**: 数学、英語、国語等
- **総得点/満点**: 例：100点中65点
- **分野別得点**: 例：数学「微分積分」20/30点、「確率」5/20点
- **ミス分類**:
  - **知識不足**: 公式を覚えていない
  - **計算ミス**: 知識はあるが実装失敗
  - **理解不足**: 問題の意図を読み違えた
  - **時間不足**: 解けたが遅い

### 2.3. 文脈データ（直近の行動）
- **連続学習日数**: 現在何日目か
- **前日のタスク完了率**: 昨日の45分は完了したか？
- **感情タグ**: テスト後「落ち込んでいる」「やる気が出た」等（任意入力）

## 3. 推論ロジック（How it works）

Claude AI（`claude-sonnet-4-6`）は、以下のステップで推論を行う。

### Step 1: 弱点の優先度付け（Matrix）
単純な「得点が低い」順ではなく、**「志望校合格に与える影響 × 改善容易性」** で優先度を決定する。

- **高優先度**: 志望校頻出分野 かつ 改善容易性が高い（例：数学の確率、英語の文法）
- **中優先度**: 志望校頻出分野 だが 改善に時間がかかる（例：数学の微積分の応用）
- **低優先度**: 志望校非頻出 かつ 既に得意（例：国語の古典文法で既に9割取れている）

### Step 2: 45分タスクの分解（Chunking）
「数学を復習する」はNG。具体的すぎるが、抽象的すぎても動けない。
**「45分で完結する最小単位」** に分解する。

- **NG**: 「確率の公式を覚える」
- **OK**: 「確率の加法定理と乗法定理の公式を暗唱し、基礎問題集P10-11の例題1-3を解く」

### Step 3: 心理的ハードルの低下（Friction Reduction）
- **連続学習日数** が長い場合：「今日は軽く復習だけ」のような低負荷タスクを提案し、継続を優先する。
- **ミス分類** が「計算ミス」の場合：知識補充ではなく、「チェックリスト作成」や「途中式の見直し」など、習慣改善タスクを提案する。
- **感情タグ** が「落ち込んでいる」場合：「この分野は全員苦戦する」という正常化メッセージを添える。

## 4. 出力フォーマット（What comes out）

ユーザーに見えるUIは、以下の3要素のみ。

### 4.1. タスクタイトル
例：「【数学】確率の基礎定着（45分）」

### 4.2. 具体的なアクションリスト
1. **0-10分**: 前回のテストで間違えた問題（P5-6）を再確認。なぜ間違えたか（知識不足/計算ミス）を赤ペンで注釈。
2. **10-35分**: 基礎問題集P10-11の例題1-3を解く。
3. **35-45分**: 正解した問題の公式を声に出して暗唱。

### 4.3. 根拠と励まし
- **根拠**: 「確率は志望校で年3問出題。基礎固めが得点力に直結します。」
- **励まし**: 「前回のテスト、計算ミスが多かったですね。今回は『確認』に特化した45分です。完璧を目指さず、完了を目指しましょう。」

## 5. 実装要件と検証計画

### 5.1. 技術スタック
- **モデル**: `claude-sonnet-4-6`（速度重視）。深い分析が必要な場合は `Opus` にフォールバック。
- **エンドポイント**: `/api/diagnose`
- **プロンプト**: `src/lib/prompts/diagnose.ts` に集約。

### 5.2. プロンプト設計の要点
- **制約**: 「45分以内で完結するタスクのみを提案すること」「抽象的な指示は禁止すること」
- **出力形式**: JSON形式で出力し、フロントエンドで整形する。

```json
{
  "task_title": "【数学】確率の基礎定着",
  "duration_minutes": 45,
  "steps": [
    {
      "time": "0-10",
      "action": "前回のテストで間違えた問題（P5-6）を再確認"
    },
    {
      "time": "10-35",
      "action": "基礎問題集P10-11の例題1-3を解く"
    },
    {
      "time": "35-45",
      "action": "正解した問題の公式を声に出して暗唱"
    }
  ],
  "rationale": "確率は志望校で年3問出題。基礎固めが得点力に直結します。",
  "encouragement": "前回のテスト、計算ミスが多かったですね。今回は『確認』に特化した45分です。完璧を目指さず、完了を目指しましょう。"
}
```

### 5.3. 3ヶ月以内の検証アクション

1. **MVP実装（2週間）**:
   - テスト結果入力フォーム（科目・点数・分野別ミス）の実装。
   - `/api/diagnose` エンドポイントの作成。
   - 固定プロンプトでJSON出力を確認。

2. **内部テスト（1週間）**:
   - 大井と相川が実際にテスト結果を入力し、出力されたタスクが「明日使えるか」を検証。
   - **検証ポイント**: タスクが具体的すぎるか、抽象的すぎないか。45分で完結するか。

3. **ベータ公開（2週間）**:
   - `[[entities/testall]]` の先行登録者（5-10名）に開放。
   - **KPI**: 提案されたタスクの完了率（目標：70%以上）。
   - **フィードバック収集**: 「このタスクは難しかった/易しかった」「時間が足りなかった」等。

4. **改善ループ（継続）**:
   - フィードバックに基づき、プロンプトの制約条件を調整。
   - 優先度付けのアルゴリズムを微調整。

## 6. リスクと対策

- **リスク**: AIの出力が「正しくない」または「非現実的」である。
  - **対策**: 出力結果を人間（大井）が確認できる仕組みを初期に導入。また、テスト結果の入力時に「この問題集を使っています」という情報を入力させ、具体的なページ番号を提案できるようにする。

- **リスク**: ユーザーが「45分」という制約を無視する。
  - **対策**: タイマー機能と連動し、残り10分で「次のアクション」を促す。また、完了報告時に「45分以内で完了しましたか？」を確認し、完了率を計測する。

## 7. 結論

このロジックにより、`[[entities/testall]]` は単なる「テスト管理アプリ」から「学習マネージャー」へ進化できる。
**証拠**: 大井自身が受験時代、復習の優先順位で迷っていた経験がある。この直感をAIに実装することで、他の受験生も同じ迷いから解放される。
**明日使う1人**: 明日テスト返却をもらい、復習に困っている高校2年生。

この設計を基に、`/api/diagnose` の実装を開始する。
