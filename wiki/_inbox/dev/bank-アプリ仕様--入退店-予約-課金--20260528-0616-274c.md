# IPAK-darts 入退店・予約・課金 仕様書 v1.0

## 1. 概要と目的

[[entities/ipak-darts]] の中核機能である「QR入退店」「予約」「課金」の仕様を定義する。
現状、事業計画は完成し物件候補も3か所あるが、**「無人オペレーション」の肝であるアプリ部分の設計が未着手**である。
月収100万円期限（2026-06-21）まで残り33日。今すぐ実装に入る必要がある。

**目的**:
1. 開発者（大井自身または相方）が迷わずコードを書けるようにする。
2. 架空の機能を作らず、既存の技術スタック（Supabase, Stripe, Next.js）で完結させる。
3. 「明日使う1人」が明確なアクションに落とし込む。

## 2. 前提条件と制約

- **ターゲット**: 1人でダーツバーに行けない潜在層（未成年含む）。
- **運用形態**: 完全無人（QR入退店・自動課金）。
- **技術スタック**:
  - フロントエンド: Next.js (App Router)
  - バックエンド/DB: Supabase (Auth, Realtime, Storage)
  - 決済: Stripe (Subscription + Payment Links)
  - ハードウェア連携: 既存のDARTSLIVEマシン（API連携はPhase2。まずは手動/QRで稼働確認）
- **禁止事項**:
  - 架空の決済会社や認証サービス名の使用。
  - 実在しないハードウェアメーカー名との連携前提設計。
  - 「AIが自動で〜」といった曖昧な記述（まずはルールベースで）。

## 3. ユーザーフロー（主要3パターン）

### 3.1 新規会員登録 → 月額課金 → 入店
1. アプリ起動 → Google/Apple認証（Supabase Auth）
2. 基本情報入力（氏名、生年月日→年齢確認、緊急連絡先）
3. 決済画面遷移（Stripe Checkout）
4. 決済完了 → 会員QRコード発行

### 3.2 入店・退店
1. 施設入口のQRリーダーでアプリQRをスキャン
2. 入店ログ記録（Supabase `check_ins` テーブル）
3. 退店時にもQRスキャン（または施設内ボタン）
4. 退店ログ記録 → 当月の残り利用回数/時間の表示

### 3.3 予約（オプション）
1. ホーム画面「予約する」をタップ
2. 日付・時間枠選択（DARTSLIVEのレーン数×稼働時間で計算）
3. 予約確定 → カレンダー登録（Google Calendar API連携はPhase2）

## 4. データモデル（Supabase）

### 4.1 `profiles` (ユーザー情報)
| カラム名 | 型 | 説明 |
|---------|-----|------|
| id | uuid | AuthユーザーID |
| full_name | text | 氏名 |
| birth_date | date | 生年月日（年齢確認用） |
| phone_number | text | 緊急連絡先 |
| created_at | timestamptz | 登録日時 |

### 4.2 `subscriptions` (課金情報)
| カラム名 | 型 | 説明 |
|---------|-----|------|
| id | uuid | PK |
| user_id | uuid | profiles.id |
| stripe_customer_id | text | Stripe顧客ID |
| stripe_subscription_id | text | Stripe購読ID |
| status | text | active, past_due, canceled |
| current_period_end | timestamptz | 課金有効期限 |
| plan_type | text | standard (4,980円), premium (7,980円) |

### 4.3 `check_ins` (入退店ログ)
| カラム名 | 型 | 説明 |
|---------|-----|------|
| id | uuid | PK |
| user_id | uuid | profiles.id |
| facility_id | text | 施設ID（1号店固定） |
| type | text | check_in, check_out |
| timestamp | timestamptz | 日時 |
| device_id | text | 端末ID（QRリーダー識別用） |

## 5. 課金設計（Stripe連携）

### 5.1 プラン構成
| プラン名 | 月額 | 内容 | Stripe Product ID |
|---------|------|------|-----------------|
| 学割 | 1,480円 | 学生証提示必須（後日検証） | `prod_student` |
| スタンダード | 4,980円 | 通い放題 | `prod_standard` |
| プレミアム | 7,980円 | マイドー保管+優先レーン | `prod_premium` |

### 5.2 実装ポイント
- **Webhook**: Stripeの `invoice.paid` イベントをSupabase Edge Functionで受信し、`subscriptions.status` を `active` に更新。
- **キャンセル**: アプリ内からStripe Portal経由でキャンセル可能にする。
- **未払い対策**: `past_due` 状態では入店QRを無効化（フロントエンド側でチェック）。

## 6. QR入退店仕様

### 6.1 QRコード生成
- ユーザーの `id` をBase64エンコードし、QRコードとして表示。
- 有効期限は30秒（セキュリティ対策）。
- Supabase Realtimeで「現在入店中」リストを更新（デモ用）。

### 6.2 QRリーダー（施設側）
- 既存のタブレットまたはスマホでQRスキャン。
- スキャン結果をSupabase `check_ins` にINSERT。
- 成功音＋画面表示「入店完了」または「退店完了」。
- **エラーケース**:
  - 未会員: 「会員登録が必要です」
  - 課金停止: 「課金状態を確認してください」
  - 重複入店: 「既に入室中です」

## 7. 予約機能（Phase1簡易版）

### 7.1 予約ロジック
- 時間枠: 30分単位（例: 10:00-10:30, 10:30-11:00）
- 最大同時予約数: レーン数（例: 5レーン）
- 予約可否: 現在の予約数 < レーン数 かつ 会員ステータス active

### 7.2 UI
- カレンダービューで空いている時間枠を緑色で表示。
- タップで予約確定。
- 予約変更/キャンセルは24時間前まで可能。

## 9. リスクと対策

| リスク | 深刻度 | 対策 |
|--------|--------|------|
| Stripeの審査で課金停止 | 高 | テスト環境で十分検証。本番はまず少額から。 |
| QRコードのなりすまし | 中 | QRの有効期限を30秒に設定。Supabaseでリアルタイム検証。 |
| 入退店ログの欠落 | 中 | 通信エラー時はローカルに一時保存。再接続時に送信。 |
| 予約のオーバーブック | 中 | SupabaseのRow Level Securityで同時アクセスを制御。 |
