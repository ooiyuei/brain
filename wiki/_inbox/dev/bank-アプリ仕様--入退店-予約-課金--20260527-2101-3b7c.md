# IPAK-darts 入退店・予約・課金仕様 v1.0

## 1. 設計思想：最小の摩擦、最大の自動化

IPAK-darts の核心は「chocoZAPのダーツ版」だが、chocoZAPと決定的に違うのは**「マシン管理の複雑さ」**と**「安全基準」**だ。

chocoZAPは自販機や簡易ゲームが中心。IPAK-dartsはDARTSLIVEなどの高価な電子ダーツマシンを扱う。
つまり、**「入退店の正確性」と「マシンの状態監視」**が命綱になる。

大井の弱点「事業多すぎ・深掘り不足」を補うため、この仕様書は**「明日から開発に入るエンジニア（または大井自身）が迷わない」**レベルの具体性を持つ。

### 基本方針
1. **QRコードの二重利用**: 入店用QRとマシン起動用QRを混同しない。
2. **課金のトリガーは「退店」**: 予約あり・なし問わず、退店時の処理で課金ロジックが動く。
3. **サブスク優先**: 月額4,980円の会員が基本。ビジターは高単価で囲い込む。

---

## 2. ユーザーフロー（3パターン）

### パターンA：月額会員（スタンダード/学割）
1. **入店**: アプリまたはWebで「入店QR」を表示 → 入口のリーダーでスキャン
2. **プレイ**: マシンのタッチパネルで「会員カード」または「QR」を読み取りプレイ開始
3. **退店**: 出口のリーダーで入店時と同じQRをスキャン → 自動退室処理
4. **課金**: 月額課金済みなら無料。超過分があれば追加課金（設定次第）

### パターンB：予約なしビジター
1. **入店**: 入口のタッチパネルまたはQRで「1時間券」を購入
2. **プレイ**: 購入した券のQRをマシンで読み取りプレイ開始
3. **退店**: 出口でQRスキャン → 時間超過分を即時クレジットカードで請求

### パターンC：予約あり（平日昼など）
1. **予約**: アプリで日時・マシン番号を指定・決済
2. **入店**: 予約QRでスキャン → マシンが予約者名でロック解除
3. **プレイ**: 通常通り
4. **退店**: 出口でスキャン → 予約枠の消化確認

## 3. ハードウェア構成（最小限の検証用）

初期は高価な専用リーダーではなく、**Androidタブレット + USB QRリーダー** で構成する。

| 場所 | デバイス | 役割 | 備考 |
|------|---------|------|------|
| **入口** | Androidタブレット + QRリーダー | 入店判定・会員確認 | 常時起動。バッテリーバックアップ必須 |
| **出口** | Androidタブレット + QRリーダー | 退店判定・課金トリガー | 同上 |
| **マシン** | DARTSLive タッチパネル | プレイ画面 | 既存マシンをそのまま利用。QR読み取りはアプリ経由 |
| **バックヤード** | PC/サーバー | 管理画面・課金処理 | Vercel/Supabase連携 |

**重要**: DARTSLiveのマシン側には改造を加えず、**「プレイ開始時に会員IDを紐付ける」**ステップを設ける。
これにより、マシンリース契約の違反リスクを回避できる。

## 4. データモデル（Supabase）

### テーブル設計

#### `profiles` (会員情報)
| カラム | 型 | 説明 |
|-------|-----|------|
| id | uuid | PK |
| email | text | ログインID |
| subscription_status | text | active / cancelled / expired |
| plan_type | text | standard / student / premium |
| current_session_id | uuid | 現在プレイ中のセッションID |

#### `sessions` (入退店履歴)
| カラム | 型 | 説明 |
|-------|-----|------|
| id | uuid | PK |
| user_id | uuid | FK -> profiles |
| entry_time | timestamptz | 入店時刻 |
| exit_time | timestamptz | 退店時刻 |
| status | text | active / completed / overstay |
| machine_ids | int[] | 使用したマシン番号リスト |

#### `machines` (マシン情報)
| カラム | 型 | 説明 |
|-------|-----|------|
| id | int | PK |
| name | text | マシン名（例: Lane 1） |
| status | text | available / occupied / maintenance |
| current_user_id | uuid | 使用中のユーザーID |

## 5. 課金ロジック（核心部分）

### 月額会員（スタンダード 4,980円/月）
- **基本**: 入退店回数無制限。
- **超過**: 1日あたり3時間超過で1,000円の追加課金（任意の設定）。
- **処理**: 退店時に `exit_time` を取得。`subscription_status` が `active` なら課金処理跳过。

### ビジター（都度払い）
- **基本**: 1時間 1,000円。
- **超過**: 15分ごとに500円の追加課金。
- **処理**: 退店時に `duration` を計算。`duration > 0` なら Stripe で即時請求。

### 予約（事前決済）
- **基本**: 1時間 800円（予約割引）。
- **処理**: 予約時にStripeで事前決済。退店時に予約枠を消化。未使用分は返金不可（規約による）。

## 6. 実装ステップ（3ヶ月以内の検証用）

### Phase 1: MVP（1ヶ月目）
- **目標**: 入退店のQRスキャンと履歴記録のみ。
- **機能**:
  - 入口/出口のQRリーダーから `entry_time` / `exit_time` をSupabaseに記録。
  - 管理画面で「現在店内人数」を表示。
- **ツール**: Supabase, Next.js, Stripe（課金なし）

### Phase 2: 課金連携（2ヶ月目）
- **目標**: Stripeとの連携でビジター課金を実装。
- **機能**:
  - 退店時にStripe Checkoutセッションを生成。
  - クレジットカード情報の登録（ビジターは初回のみ）。
- **ツール**: Stripe API, Next.js

### Phase 3: 予約・サブスク（3ヶ月目）
- **目標**: 月額会員の管理と予約機能。
- **機能**:
  - Stripe Subscriptions の連携。
  - 予約カレンダーの実装。
  - 入口での会員QRスキャン時、サブスク有効性のリアルタイムチェック。
- **ツール**: Stripe Webhooks, Next.js

## 7. リスクと対策

### リスク1: QRの不正利用（写真共有など）
- **対策**: QRコードにタイムスタンプを含め、5分ごとに更新する。
- **追加**: 入口と出口で同じQRが使われた場合、アラートを出す。

### リスク2: マシンの故障
- **対策**: マシンの状態を `maintenance` に設定すると、予約・プレイ不可にする。
- **追加**: 故障報告ボタンを管理画面に設置。

### リスク3: 課金の失敗
- **対策**: 退店時の課金失敗時は、強制的に「未払い」として次回入店をブロック。
- **追加**: 管理者が手動で課金処理できる機能。
