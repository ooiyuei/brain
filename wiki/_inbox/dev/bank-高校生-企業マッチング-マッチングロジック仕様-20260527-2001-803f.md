# EEMUS マッチングロジック仕様 v1

静岡の高校生と地元企業をつなぐ「高校生版Wantedly」の核心は、**「見えない適性」と「見えない課題」の翻訳**にある。

Wantedlyが「スキル・経験」でマッチングするなら、EEMUSは**「潜在能力・地域愛・学習意欲」**でマッチングする。
既存の求人票（職務記述書）は「過去の実績」しか見せない。
EEMUSは「未来の可能性」を見る。

## 1. 基本設計思想：3層マッチング

単純なキーワード一致（例：建設業 ∧ 静岡）では、ミスマッチと早期離職を防げない。
以下の3層でフィルタリングする。

| レベル | 名称 | 判定ロジック | 目的 |
|--------|------|------------|------|
| L1 | **ハード制約** | 居住地・通学圏・業種・雇用形態 | 物理的に可能か |
| L2 | **ソフト適合** | 適性診断結果 × 企業カルチャー | 長く働けるか |
| L3 | **共創ポイント** | 学生の興味 × 企業の課題 | 熱中できるか |

**L1は排除ロジック、L2とL3は推薦ロジック。**

## 2. 学生側データ構造（Input）

学生が入力・収集するデータは「履歴」ではなく「特性」に絞る。

### 2.1 必須属性（L1用）
- **居住地**: 最寄駅・通学時間許容範囲（例：電車で1時間以内）
- **学年**: 1年（体験重視）/ 2年（インターン候補）/ 3年（就職直結）
- **希望業種**: 複数選択可（建設、製造、IT、飲食、福祉など）
- **雇用形態**: 短期体験（1日〜）、長期インターン、就職希望

### 2.2 適性・特性データ（L2用）
EEMUS独自の簡易適性診断（5問程度）で取得。
既存のSPIや適性検査は「正解」を求めるが、EEMUSは「傾向」を見る。

| 項目 | 質問例（意図） | 分類 |
|------|--------------|------|
| **問題解決スタイル** | 「道に迷った時、どうする？」<br>A: 地図で確認する（論理型）<br>B: 人に聞く（対人型）<br>C: 周囲を見て推測する（察し型） | 思考傾向 |
| **チーム役割** | 「グループ作業で自分は？」<br>A: 進行役（まとめ役）<br>B: 資料作成（裏方）<br>C: アイデア出し（発案） | 役割適合 |
| **価値観** | 「仕事で一番大事なのは？」<br>A: 安定・確実性<br>B: 成長・挑戦<br>C: 人間関係・雰囲気 | 文化適合 |
| **地域愛指数** | 「静岡で暮らすことについて？」<br>A: 好き/住み続けたい<br>B: 特に考えない<br>C: 出たい | 定着性 |
| **学習意欲** | 「新しいことを学ぶ時？」<br>A: 本で調べる<br>B: 人に教わる<br>C: やりながら覚える | 教育コスト |

**データ活用**:
- 「対人型」かつ「人間関係重視」→ 接客・営業・教育系企業へ推薦
- 「論理型」かつ「確実性重視」→ 製造、経理、インフラ系へ推薦
- 「察し型」かつ「成長重視」→ 企画、マーケティング、スタートアップ系へ推薦

### 2.3 興味・共創ポイント（L3用）
- **興味キーワード**: 自由入力（例：「ロボット」「料理」「環境問題」「SNS」）
- **やってみたいこと**: 複数選択（例：「現場を見る」「実際に触る」「提案する」「数字を見る」）
- **持ち味**: 自由入力（例：「図を描くこと」「話を聞くこと」「整理すること」）

## 3. 企業側データ構造（Input）

企業が入力するデータは「求人票」ではなく「受け入れ体制・カルチャー」。

### 3.1 基本情報（L1用）
- **業種**: 建設、製造、小売、サービス、ITなど
- **所在地**: 最寄駅・アクセス
- **受け入れ可能数**: 1名〜
- **受け入れ期間**: 1日体験、1週間、1ヶ月以上など
- **対象学年**: 1年〜3年

### 3.2 カルチャー・環境データ（L2用）
企業アンケートで取得。

| 項目 | 質問例 | 分類 |
|------|--------|------|
| **職場の雰囲気** | 「静かに集中する」「活発に話す」「個人作業が多い」 | 環境適合 |
| **指導スタイル** | 「マニュアル通り」「現場で見て覚える」「 mentor制」 | 教育適合 |
| **主な業務内容** | 「顧客対応」「データ入力」「機械操作」「企画立案」 | 業務適合 |
| **大切にしていること** | 「正確さ」「スピード」「創意工夫」「チームワーク」 | 価値観適合 |

### 3.3 共創ポイント（L3用）
- **学生にやってほしいこと**: 「新しい視点を持ってほしい」「若者の声を聞きたい」「人手不足を補ってほしい」
- **期待する成果**: 「職場の雰囲気向上」「業務効率化のアイデア」「単純作業の補助」

## 4. マッチングアルゴリズム（ロジック）

### 4.1 スコアリング方式

各マッチングは **0-100点** で評価。

#### L1: ハード制約（通過/不通過）
- 居住地が通学圏外 → **0点（除外）**
- 業種が希望外 → **0点（除外）**
- 学年が受け入れ対象外 → **0点（除外）**
- すべてOK → **100点（通過）**

#### L2: ソフト適合（重み付き合計）
- **思考傾向適合度**: 学生の問題解決スタイルと企業の業務内容の相関（例：論理型 × データ処理 = 高適合）
- **役割適合度**: 学生のチーム役割と企業の期待する貢献の相関
- **価値観適合度**: 学生の価値観と企業の大切にしていることの一致度
- **地域愛指数**: 地域愛が高い学生には、地元企業にボーナス加点（定着率向上のため）

**計算式**:
```
L2スコア = (思考傾向適合度 × 0.3) + (役割適合度 × 0.3) + (価値観適合度 × 0.3) + (地域愛ボーナス × 0.1)
```
※重みは後期検証で調整。初期は均等か、地域愛を少し重くする。

#### L3: 共創ポイント（キーワード一致度）
- 学生の「興味キーワード」と企業の「期待する成果」のキーワードマッチング
- 例：学生「ロボット」 × 企業「自動化に興味」 → 高適合
- 例：学生「料理」 × 企業「接客」 → 低適合（業種は合っても共創ポイントは低い）

**計算式**:
```
L3スコア = キーワード一致数 × 重み係数
```

### 4.2 総合スコアと推薦順位

```
総合スコア = L1(通過フラグ) × (L2スコア × 0.7 + L3スコア × 0.3)
```

- **L1不通過**: 推薦リストから除外
- **L1通過**: L2とL3の加重平均で順位付け
- **トップ3企業**を学生に提示
- **トップ3学生**を企業に提示（匿名化済みプロフィール）

## 5. 出力形式（UI/UX）

### 学生側画面
- **「あなたに合う企業 TOP3」**
    - 企業名（匿名化可：「静岡市にある建設会社A」）
    - **マッチ理由**: 「あなたの『整理する力』と、同社の『データ管理強化』が合致します」
    - **共創ポイント**: 「業務効率化のアイデアを提案できます」
    - **カルチャー適合度**: 85%
    - **アクション**: 「見学予約」「質問送信」

### 企業側画面
- **「あなたに合う学生 TOP3」**
    - 学生名（匿名化可：「静岡市在住 高校2年生」）
    - **適合理由**: 「地域愛指数が高く、長期的な関与が期待できます」
    - **適性プロフィール**: 「対人型・アイデア出し役・学習意欲高」
    - **共創ポイント**: 「新しい視点での業務改善アイデアが期待できる」
    - **アクション**: 「見学を誘う」「面接予約」

## 6. 初期フェーズの制約と対応

### 6.1 データ不足への対応
- **初期**: 学生・企業ともにデータが少ない。
- **対応**:
    - 学生には「適性診断」を必須化（5問で完了）
    - 企業には「カルチャーアンケート」を必須化（3問で完了）
    - キーワードマッチングを重視（L3スコアを高く設定）

### 6.2 誤マッチングの防止
- **初期**: アルゴリズムが未熟。
- **対応**:
    - **人間によるオーバーレイ**: 初期はスタッフ（EEMUSチーム）がマッチ結果を確認し、微調整する
    - **フィードバックループ**: 見学後の評価（相性）をデータとして蓄積し、アルゴリズムを改善

## 7. 将来の拡張

- **AIによるマッチング精度向上**: 見学後の評価データを学習させ、適合度を高精度化
- **キャリアパスの可視化**: 見学→インターン→就職の流れをデータ化し、学生の成長軌跡を表示
- **地域経済への影響分析**: 地元企業への就職率向上と、学生の地域定着率の相関を分析
