# 会社別AI保存庫コア

> AIpaXの中核フレームワーク。顧客企業ごとの情報基盤として作り、その上に複数の業務AIツールを追加していく。

## 構造

```
会社別AI保存庫コア（マザーボード）
+
顧客ごとの業務AIツール（拡張機能）
```

## 何のためか

単に「資料をフォルダに置く」だけでは起きる問題：

| 問題 | 例 |
|------|----|
| 料金表が複数あり最新版不明 | 2024年版と2026年版が混在 |
| 契約書とFAQで内容が違う | 解約条件の表現がズレている |
| 顧客ごとに特別条件がある | A社だけ無料延長 |
| 旧プランと新規で条件が違う | レガシー客の優遇 |
| 返金条件が資料にない | 顧客が直接交渉した経緯のみ |
| 通話ログの話が正式情報か不明 | AIが拾ってFAQ化していいか分からない |

保存庫コアは、これらを **AIが扱える形** に整理する。

## 保存庫コアに入れるもの

会社の **正式情報・元資料・根拠・履歴**：
会社情報 / 料金表 / 契約書 / 利用規約 / 解約条件 / 返金条件 / FAQ / 商品サービス情報 / 営業資料 / サポート範囲 / 顧客情報 / 顧客別条件 / 例外条件 / 禁止表現の原本 / エスカレーション条件の原本 / 過去問い合わせ / 過去商談メモ / メール本文 / 通話ログ元データ / 情報の有効期間 / バージョン / 更新履歴 / 確認履歴 / 情報同士の関係

## 保存庫コアに入れないもの

**ツール出力・AI推測**：
AI回答候補 / 通話中の回答カード / 検知された質問 / 営業トーク案 / 議事録要約 / 次アクション案 / フォロー文案 / 管理レポート / AIの推測 / AIが勝手に作ったFAQ候補 / 改善提案

→ 保存庫コアに混ぜると **保存庫が汚れる**。原則として：

```
通話ログ → ツール側ログ → 入庫前処理 → 確認事項化 → 顧客確認 → 保存庫コアへ正式反映
```

## データ構造

| テーブル | 役割 |
|---------|------|
| **companies** | 顧客企業情報（会社名/業種/事業内容/対象領域/契約状態） |
| **sources** | 元資料・元ログ（料金表PDF/契約書/FAQ/営業資料/メール/通話文字起こし/顧客別契約書） |
| **source_chunks** | 検索しやすく分割した本文 |
| **knowledge_items** | 資料から抽出した知識単位 |
| **entities** | 登場するもの（顧客/担当者/商品/サービス/プラン/契約/部署/プロジェクト） |
| **relationships** | 情報同士の関係（この料金はこのプランに紐づく等） |
| **events** | 起きたこと（契約/資料更新/メール送信/問い合わせ/質問/承認） |
| **reviews** | 確認履歴 |
| **versions** | 変更履歴・置き換え関係 |

## 入庫前処理フロー

顧客資料を保存庫コアに入れる前にAIで検査・分類・抽出：

```
1. 顧客から資料受領（まだ保存庫に入れない）
2. DBに source として登録
   （source_id/company_id/source_type/file_name/title/created_at/updated_at/status）
3. 資料をチャンク化（料金表ならプランA/B/初期費用/解約条件/注意事項に分割）
4. AIで knowledge_candidate 抽出
   （category/subject/content/source_id/review_status=unreviewed）
5. 矛盾・不足・最新版不明を検出 → 確認項目作成
6. 顧客確認シートに出す
7. 顧客OK分のみ正式 knowledge_item として保存庫コアへ
8. 検索用インデックス作成
   （company_id/is_active/effective_from/effective_until/review_status/version/対象領域で絞る）
```

**重要：AIが勝手に補完しない**。
- 不明 → 不明として出す
- 矛盾 → 矛盾として出す  
- 不足 → 不足として出す

## 顧客確認シートの形

| 項目ID | 種別 | 抽出内容 | 根拠資料 | 問題種別 | 確認してほしいこと | 顧客回答欄 | 正式反映OK/NG | 備考 |
|--------|------|---------|---------|---------|-----------------|----------|-------------|------|

## 責任分担

### 顧客側責任
- 正しい資料を提出する
- どれが最新版か決める
- 矛盾資料の正誤を決める
- 足りない資料を出す
- 正式ルールを決める
- 確認事項に回答する
- 保存庫反映の承認
- 通話相手への録音・文字起こし通知方針
- 顧客側アカウント管理にする場合の権限提供

### こちら側責任
- 資料を解析する
- 不足を検出する
- 矛盾を検出する
- 最新版不明を検出する
- 確認事項を作る
- 確認済み情報を保存庫に反映する
- 保存庫を検索可能にする
- 業務ツールと接続する
- 月額内の軽微更新・調整

→ 顧客側が確認事項に返事しない・資料矛盾・最新版不明・資料出てこない場合は **止まる**（顧客側責任）

## 通話ログ・AI要約の扱い

顧客と最初に決める：
- 通話ログ保存するか / 音声保存するか / 文字起こし保存するか
- 保存期間
- AI要約保存するか / 形式 / 自動保存OK項目 / 顧客確認必要項目

合意できれば、**通話要約・質問一覧・未回答質問・次回確認事項・顧客の関心は自動保存OK**。

ただし：
- 料金 / 契約条件 / 返金条件 / 解約条件 / 顧客別条件 / FAQ / 正式ルール → **顧客確認必須**

```
元ログ → 保存方針に従って保存
AI要約・整理ログ → 事前合意の形式で自動保存
正式情報候補 → 確認待ち
正式情報 → 顧客確認後に保存庫コアへ反映
```

## 技術構成

| レイヤー | 採用候補 |
|---------|--------|
| 構造化DB | **Supabase**（companies/sources/source_chunks/knowledge_items/entities/relationships/events/reviews/versions/tool_logs/user_accounts/permissions） |
| 意味検索 | **OpenAI File Search / Embedding** |
| アプリ | URL+ログイン（初期）。営業担当画面/管理者画面/確認待ち項目画面/資料一覧画面/利用ログ画面 |

※デスクトップアプリ化は初期では不要。

## OpenAI/Supabase 管理方式

顧客に選択肢を提示：

| 方式 | メリット | デメリット |
|------|--------|----------|
| A. 自社側管理（標準） | 導入早い・顧客負担少ない・修正しやすい | 顧客データを当方基盤で扱う説明必要 |
| B. 顧客側管理 | データ基盤を自社保持・機密に強い | 設定・権限・運用重い・導入遅い |
| C. 自社→顧客側移管 | 早く導入後で移管 | 移管作業は別料金 |

## 5原則

1. **保存庫コアとツール出力を混ぜない**
2. **顧客資料の正しさは顧客側が決める**
3. **AIが勝手に正式情報を作らない**
4. **不足・矛盾・最新版不明を検出して提示する**
5. **対象領域内の顧客別条件・例外条件は削らない**

## 拡張ツール候補

保存庫コアの上に載せるツール：

| ツール | 顧客の課題 |
|--------|----------|
| リアルタイム電話/営業回答支援 | 営業電話で回答がズレる |
| 社内AIチャット/新人教育AI | 新人がルールを何度も聞く |
| 議事録整理AI | 商談後の整理が遅い |
| フォロー文生成AI | フォロー漏れ多い |
| CS対応AI | 問い合わせ属人化 |
| 管理レポートAI | 管理者が現場見たい |
| CRM補助ツール | 既存CRMが使われない |

**最有力の初手商品**：保存庫コア + リアルタイム電話/営業回答支援
