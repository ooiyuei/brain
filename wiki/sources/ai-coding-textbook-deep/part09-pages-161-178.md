## セクション別詳細

### 1. メンション機能の応用 (p.161)

#### スタイル参照のメンション
プロンプト例:
```
@main.js を参考に、メッセージをコピーするボタンを追加してください
```
→ AIは main.js の書き方やスタイルを参考にして同じ雰囲気のコードで新機能を追加。

#### 複数ファイルメンション
```
@index.html @style.css チャット画面のレイアウトを改善してください
```
→ HTMLとCSSの両方を考慮した修正を提案。

#### フォルダメンション
```
@docs どんなファイルがあるかを分析して説明してください
@src 内のすべてのファイルで、コメントを日本語に統一してください
```
- Antigravity では対応
- ツールによって対応状況が異なる旨を明記

#### メンションのコツ (p.162)
- **必要最小限**: 関係ないファイルを大量にメンションするとAIが混乱・処理遅延
- **量が多すぎる場合はタスク分割**: 10ファイル一括ではなく、5ファイル+5ファイルに段階分け

- Antigravity: `@web`

### 3. Webサーチの注意点・Secure Mode (p.164)

#### プロンプトインジェクションのリスク
- Antigravityはデフォルトで許可なしで作業を進める
- AIがWebから「悪意のある指示」を読み込むリスク (隠された指示)

#### Secure Mode 有効化手順
```
メニューバー → Antigravity → Preferences → Antigravity Settings → Secure Mode
```
- AIが作業を進める前に逐一確認を求める
- 利便性は下がるが、初学者は安全性優先で

#### 追加対策
- 無闇にAIにWeb検索をさせない
- 検索結果は必ず自分で確認してから使う
- 後述ガードレールセクションで紹介する **AllowList** (許可リスト) でドメイン絞り込み可能

### 4. MCP: 能力を拡張する (p.164-165)

#### MCPとは
- **Model Context Protocol** の略
- AIエージェントに新しい能力を追加するための仕組み
- 様々なツール・サービスをAIエージェントに接続するための共通規格
- 比喩: 「AIの世界における USB-C」
- 出典: https://norahsakal.com/blog/mcp-vs-api-model-context-protocol-explained/

#### MCPで追加できる能力 (4分類)
1. **ファイル操作**: パソコン内のファイル読み書き
2. **Web操作**: ブラウザを自動で動かして情報取得
3. **データベース操作**: データベースへの問い合わせ
4. **外部サービス連携**: カレンダー・メール・クラウドストレージ等

### 5. MCP導入ハンズオン: Context7 (p.165-171)

#### Antigravityでの導入手順

**Step 1**: チャット右上の「・・・」→ MCP Servers をクリック
**Step 2**: デフォルト一覧は開発者向けに偏っている → 「Manage MCP Servers」→「View raw config」

#### Context7とは (p.166)
- AIがライブラリ使用時に最新ドキュメントを参照できるようにするMCPサーバー
- AIの知識が古いことに起因する「古めかしいコード」「動かないコード」を防ぐ

#### Step 3: APIキー取得 (p.167)
- Sign In → Googleログイン等でアカウント作成
- 公開リポジトリ参照分は無料
- 「Create API Key」をクリック → API キーに名前を付けて保存

#### Step 4: Antigravity設定ファイル記述 (p.168)
```json
{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@upstash/context7-mcp", "--api-key", "YOUR_API_KEY"]
    }
  }
}
```
→ Manage MCP Servers からリフレッシュボタン → Context7 が出現

#### Step 5: ルール追加で自動起動化 (p.169-170)
- 「Customizations」→「+Workspace」
- ルール名: `use-context7-for-docs`
- 公式推奨ルール (日本語訳):
  > 「コード生成、セットアップや設定手順、またはライブラリ/APIドキュメントが必要な場合は、常に Context7 を使用してください。つまり、私が明示的に依頼しなくても、Context7 MCPツールを自動的に使用してライブラリIDを解決し、ライブラリのドキュメントを取得する必要があります。」
- 出典: https://context7.com/docs/installation#requirements

#### 実践: Context7の効果検証 (p.171)
- 本書執筆時点 (2025年12月)、Gemini 3 が1ヶ月前にリリース
- プロンプト:
  ```
  Gemini SDKで使える最新のモデルって何？
  ```
- 動かない場合は末尾に `@context7` を追記
- AntigravityがContext7経由でGemini SDKドキュメントを検索→「Gemini 3」を回答

### 6. MCPの注意点 (p.172)

#### 権限リスク
- MCPサーバーをインストール = AIにその機能を使う権限を与える
- 悪意あるMCPサーバーインストール → AIを通じてPCが悪用される可能性

#### 安全性
- **基本安全**: Anthropic公式 / 信頼できる企業提供
- **要慎重**: 野良MCPサーバー (個人公開)

#### 確認すべき4ポイント
1. どんな機能を提供しているのか?
2. どのファイルやサービスにアクセスするのか?
3. 提供元は信頼できるか? (GitHubスター数、作者の実績)
4. 本当に必要なMCPサーバーだけインストール

#### 著者個人の感覚
> 「個人が作成しているMCPサーバー原則インストールしないぐらいの感覚で良い」

### 7. ルールファイル: AI への作業マニュアル (p.172-174)

#### 課題提起
- 毎回「です・ます調で書いてください」「箇条書きを使ってください」を繰り返すのが面倒

#### ルールファイルとは (p.173)
- AIエージェントに読み込ませる「作業マニュアル」
- ファイル名規約:
  - **AGENTS.md** (多くのAIコーディングエージェント) — 公式: https://agents.md/
  - **CLAUDE.md** (Claude Code)
- タスク実行前にAIが読み込んで取り掛かる

#### ⚠️ 警告
> 「ルールファイルは100%読み込まれるわけではありません。内容が肥大化しすぎると、一部が無視されたり、重要なルールが見落とされたりすることがあります。」
- 効果的な書き方は第9章で詳述

#### ルールファイル記述例 (p.173)
```markdown
# プロジェクト概要

これは業務報告書と議事録を作成するためのプロジェクトです。

## 文体・トーン

- 「です・ます調」で統一する
- 専門用語は最初に必ず説明を入れる
- 箇条書きを積極的に活用して読みやすくする

## 週次報告書の構成

1. 今週の実績 (3-5項目)
2. 来週の予定 (3-5項目)
3. 課題・懸念事項 (あれば)
```

#### 効果
- 「週次報告書を作って」だけでAIが自動的にこの構成で作成
- 毎回詳しく指示する必要がなくなる
- AIを「育てる」方法は第9章で詳述

### 8. カスタムコマンド / ワークフロー (p.174)

#### 概要
- 定型作業を一言で実行できる便利機能
- 専用Markdownファイルとして作成
- ファイル名がそのままコマンド名 (例: `proofread.md` → `/proofread`)

#### 校閲コマンド実装例
```markdown
---
description: 文章の校閲を行う
---

## タスク

以下の観点で文章を校閲してください。

- 誤字脱字のチェック
- 表記ゆれの統一 (「〜できる」と「〜出来る」など)
- 文末表現の統一 (です・ます調の混在がないか)
- 不自然な表現や冗長な箇所の指摘
- 修正箇所は「修正前 → 修正後」の形式で提示
```

→ `/proofread` とタイプするだけで実行

### 9. Antigravity 図解自動生成ワークフロー実装 (p.175-177)

#### 目的
- 選択したテキスト内容を解説する図解画像を生成
- ドキュメントに自動挿入

#### 設定手順 (p.175)
1. チャット欄右上「・・・」→「Customizations」
2. 「Workflows」タブに切り替え
3. 「Workspace」をクリック
4. ワークフロー名: `add-image`
5. Description (説明) と Content (実際の指示) を記入

#### ワークフロー内容例 (p.176)
```
与えられたテキストを解説する図解を作成する

1. チャットで与えられたテキストに基づき、Nano Banana Proで図解画像を生成します。
2. 生成画像を内容に近い、適切な位置に保存します (例: `feature_summary.png`)
3. 与えられた画像を内容を表す英語の名称にスタイリングに保存します
   - 白基調とした表現するこのインクされた画像のリンクを追加
   - ファイル名は必ず内のプロフェッショナルなディレクトリに保存
   - 画像はマークダウンに保存します
```
※注: PDFテキスト化での乱れあり。「Nano Banana Pro」「英語ファイル名」「マークダウンへのリンク追加」が要点。

#### 実行方法
1. テキストファイルを開く
2. 図解を生成したい箇所を選択
3. `Ctrl + L` (Windows) または `Command + L` (Mac)
4. チャット欄で `/add-image` 入力 → 送信

#### 効果
> 「一回ワークフローを設定すると、毎回同じ指示を繰り返す必要がなくなるため、作業効率が大幅に向上します。」

### 10. なぜファイル名・コマンド名を英語にするのか (p.178)

#### 理由1: 文字化け回避
- Windows と Mac で日本語ファイルをやり取り → ファイル名が正しく表示されない
- 英語 (ASCII文字) なら、どんな環境でも確実に動く

#### 理由2: スペースの扱い
- 日本語: 「週次 報告書.txt」のようにスペースを入れがち → コマンドラインで面倒
- 英語: `weekly-report.txt` のようにハイフン・アンダースコアで繋げるのが自然

#### 結論「状況による」
**個人で使うだけなら日本語でも OK**

**英語を推奨する3ケース**:
1. AIコーディングツールで扱う場合 (カスタムコマンドなど)
2. 他の人と共有する可能性がある場合
3. Gitなどでバージョン管理する場合

#### リスク
- AIエージェントは時に予期しないコマンドを実行しようとする
- 人間の指示を解釈して動くが、誤解や想定外の方法で問題解決を試みることも

## 💡 Tips・原則

### メンションの3原則
1. **必要最小限**: 関係ないファイル大量メンションはAI混乱・処理遅延の原因
2. **段階分割**: 10ファイル一括ではなく5+5のような分割で進める
3. **フォルダメンション対応はツール依存**: Antigravityは対応、他は要確認

### Webサーチの安全運用
- **デフォルト動作はリスキー**: Antigravityは許可なし作業 → プロンプトインジェクション可能性
- **Secure Mode**: 初学者には必須
- **AllowList**: ドメイン絞り込みでリスク軽減
- **検索結果は必ず人間が確認**

### MCP導入の安全チェックリスト
- ☐ どんな機能を提供しているか?
- ☐ どのファイル/サービスにアクセスするか?
- ☐ 提供元は信頼できるか? (GitHubスター数, 作者実績)
- ☐ 本当に必要なMCPサーバーか?
- ☐ 個人作成の野良サーバーは原則回避

### ルールファイル運用の3原則
1. **100%読み込み保証はない**: 肥大化に注意
2. **共通ルールのみ記述**: タスク固有詳細は別ファイル
3. **次章 (第9章) で詳細展開**

### カスタムコマンドの命名規則
- ファイル名 = コマンド名
- 英語推奨 (AIツール内, 共有可能性, Git管理時)
- 個人利用のみなら日本語OK

### Context7 統合のコツ
- ルールファイルに自動起動指示を書く → `@context7` 明示不要に
- ライブラリ・APIドキュメント参照タスクで自動発火
- AI知識の古さ問題を根本解決
