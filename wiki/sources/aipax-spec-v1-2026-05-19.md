# AIコンサル事業（AIpaX）共有資料 v1

> 大井から共有された最新版仕様書。**既に4社契約取れている**前提で、追加3社獲得＋既存4社のオンボーディング・拡張をぶん回す。
> この仕様書は顧客向け説明資料の母体。営業文・LP・契約書・提案書を生成する時の **根拠**。

## 🔥 ステータス: 4社契約済 (2026-05-19時点)

- 初回事例価格レンジで4社獲得
- 次の目標: 5/26時点で追加3社接触・1社クロージング
- 既存4社: オンボーディング+月額運用へ移行
- 単価アップ: 5社目以降は通常価格(初期100-150万・月額10-15万)

## 0. 共有目的

会社別AI保存庫コア+業務ツール の事業モデル全体共有。
電話支援は最初の商品で、保存庫コアは汎用基盤。

## 1. 全体構想

**会社別AI保存庫コア + 顧客ごとの業務AIツール**

保存庫コア = マザーボード
各AIツール = 拡張機能

業務ツール例:
- リアルタイム電話/営業回答支援
- 社内AIチャット
- 新人教育AI
- 議事録整理AI
- フォロー文生成AI
- CS対応AI
- 管理レポートAI
- CRM補助ツール

## 2. 売り方の基本形

**保存庫コア構築 + 最初の業務ツール1つ**

| 顧客課題 | 最初に提案するツール |
|---|---|
| 営業電話で回答がズレる | リアルタイム電話/営業回答支援 |
| 新人が社内ルール何度も聞く | 社内AIチャット/新人教育AI |
| 商談後の整理が遅い | 議事録整理AI |
| 問合せ対応が属人化 | CS対応AI |
| フォロー漏れ多い | フォロー文生成AI |
| 管理者が現場見たい | 管理レポートAI |

現時点の主力: **保存庫コア + リアルタイム電話/営業回答支援**

## 3. 会社別AI保存庫コアとは

会社情報を AIが扱える形に整理する基盤。

管理する情報状態:
- どの資料由来か
- どの情報が正式か
- どの情報が古いか
- いつから/いつまで有効か
- どの顧客に適用されるか
- どの条件で例外になるか
- どの情報同士が関係しているか
- どの情報は確認済み/未確認か

解決する問題:
- 料金表が複数あり、どれが最新か不明
- 契約書とFAQで内容が違う
- 顧客ごとに特別条件がある
- 旧プラン顧客と新規顧客で条件が違う
- 返金条件が資料に書かれてない
- 営業資料には書いてあるが正式ルール不明
- 通話ログの話が正式情報か不明

## 4. 保存庫コアに入れるもの

会社の正式情報・元資料・根拠・履歴。

主要テーブル:
- **companies** — 顧客企業情報
- **sources** — 元資料/元ログ/元ファイル
- **source_chunks** — 検索しやすい分割本文
- **knowledge_items** — 資料から抽出した知識単位
- **entities** — 登場する物(顧客/担当/商品/プラン/契約/部署)
- **relationships** — 情報同士の関係
- **events** — 起きたこと
- **reviews** — 確認履歴
- **versions** — 変更履歴/置き換え関係

## 5. 保存庫コアに入れないもの

- AIの回答候補
- 通話中表示の回答カード
- 検知された質問
- 営業トーク案
- 議事録要約
- 次アクション案
- フォロー文案
- 管理レポート
- AIの推測
- AIが勝手に作ったFAQ候補
- ツール出力そのもの

→ これらは ツール側ログ として別管理。
→ 顧客確認を通った後だけ正式情報化。

## 6. 入庫前処理

顧客資料を保存庫コアに入れる前のAIによる検査・分類・抽出:

- 資料種別判定
- チャンク化
- 知識候補抽出
- 根拠箇所紐付け
- 不足情報検出
- 矛盾情報検出
- 最新版不明検出
- 有効期間不明検出
- 保存可能/保留に振り分け
- 顧客確認シート作成

**重要原則: AIが勝手に補完しない**。不明=不明、矛盾=矛盾、不足=不足 として出す。

## 7. 顧客責任 vs こちら側責任

### 顧客側責任
- 正しい資料提出
- 最新版判定
- 矛盾資料の正誤判定
- 不足資料提出
- 正式ルール決定
- 確認事項への回答
- 保存庫反映の承認
- 通話相手への通知方針決定
- (顧客側管理の場合) 必要権限提供

### こちら側責任
- 資料解析
- 不足/矛盾/最新版不明検出
- 確認事項作成
- 確認済情報を保存庫に反映
- 検索可能化
- 業務ツールとの接続
- 月額内軽微更新/調整

→ 顧客が確認事項に返事しない・資料矛盾・最新版不明・資料未提出の場合は そこで止まる。顧客側責任。

## 8. 入庫処理フロー

1. 顧客から資料を受領
2. Supabase等に source として登録
3. 資料をチャンク化 (source_chunks)
4. AIで知識候補抽出 (knowledge_candidate)
5. 矛盾/不足/最新版不明を検出
6. 顧客確認シート作成
7. 顧客OK出たもののみ正式反映 (knowledge_item)
8. 検索インデックス作成 (company_id/is_active/effective_from/until/review_status/version で絞る)

## 9. 技術構成

### Supabase等のDB
構造化データ保管。
- companies/sources/source_chunks/knowledge_items
- entities/relationships/events/reviews/versions
- tool_logs/user_accounts/permissions

### OpenAI/File Search/Embedding
資料検索・意味検索。
**検索結果だけで勝手に正式情報化しない**。

### アプリ画面
初期: URL+ログインで充分。
- 営業担当画面
- 管理者画面
- 確認待ち項目画面
- 資料一覧画面
- 利用ログ画面

## 10. リアルタイム電話/営業回答支援 (最初のツール)

### 何をするか
営業/電話対応中に相手の質問をリアルタイム検知 → 保存庫検索 → 営業担当に回答候補表示。

### 流れ
通話音声
→ リアルタイム文字起こし
→ 質問検知
→ 保存庫コア検索
→ 回答候補生成
→ 根拠表示
→ 注意事項表示
→ 営業担当が見て答える

### 表示要素
- 検知された質問
- 回答候補
- 根拠資料
- 注意事項
- 言ってはいけないこと
- 確認必要事項
- エスカレーション条件

### 初期範囲に含む
- PCブラウザマイク入力
- リアルタイム文字起こし
- 質問検知
- 保存庫検索
- 回答カード表示
- 根拠/注意事項/言ってはいけないこと/確認必要 表示

### 初期範囲外
- AIが顧客へ直接音声返答
- 電話回線連携
- CRM連携
- Zoom/Meet連携
- 通話後議事録
- 管理レポート
- 全通話FAQ自動生成
- 法務判断代行

## 11. 通話ログ・AI要約の扱い

事前合意で決めること:
- 通話ログ保存するか
- 音声保存するか
- 文字起こし保存するか
- 保存期間
- AI要約保存するか
- 要約形式
- 自動保存OK項目
- 顧客確認必要項目

### 区分
- 元ログ → 保存方針に従って保存
- AI要約/整理ログ → 事前合意した形式で自動保存
- 正式情報候補 → 確認待ち
- 正式情報 → 顧客確認後に保存庫反映

→ 通話ログ・AI要約をそのまま正式情報にしない。
→ 毎回すべて確認に回す必要もない。

## 12. OpenAI/Supabase 管理方式 (顧客選択)

### A. 自社側管理 (標準案)
- メリット: 導入早・顧客負担少・改善しやすい
- デメリット: 説明必要・実費管理

### B. 顧客側管理 (追加費用)
- メリット: 顧客がデータ基盤を自社所有・機密情報強い
- デメリット: 設定/権限/運用が重い・導入遅い

### C. 自社側→顧客側へ移管 (折衷)
- 最初自社側で素早く・固まったら移管
- 移管作業は別料金

## 13. 通話ログ保存 (顧客選択)

### A. 保存しない
- メリット: セキュリティ説明軽い
- デメリット: 改善検証使いにくい

### B. 音声なし・文字起こし/検知質問/回答候補/検索ログだけ一定期間保存 (標準)
- メリット: 改善使える・音声残らず
- デメリット: 保存期間/削除ルール要定義

### C. 音声も文字起こしも保存 (追加設計)
- メリット: 後から確認・教育・品質管理
- デメリット: 同意/期間/削除/データ管理の説明重い

## 14. 録音・文字起こし同意

顧客側責任で運用。
こちらは通知文言案・運用ルール案を提供。

例:
「この通話は、対応品質向上および回答支援のため、文字起こしされる場合があります」

## 15. 価格感

### 初回事例価格 (現在のレンジ・4社契約済)
- 初期: 50万〜80万円
- 月額: 5万円前後

### 通常価格 (5社目以降目標)
- 初期: 100万〜150万円
- 月額: 10万〜15万円以上

### 本格導入
- 初期: 200万〜500万円
- 月額: 20万〜50万円

価格決定要因 (資料枚数ではない):
- 管理対象領域
- 更新頻度
- ツール数
- 利用量/実費
- 外部連携
- 運用対応速度
- 紙資料/OCR
- 大量旧資料整理
- 自社側 vs 顧客側管理

→ 対象領域内なら顧客別条件・例外条件を削らない。それが保存庫の価値。

## 16. 月額でやること

- 対象領域内の資料更新
- 入庫前処理
- 顧客確認後の保存庫反映
- 旧情報の非アクティブ化
- 検索条件の軽微調整
- 回答カードの軽微改善
- 利用ログ確認
- 回答できなかった質問の整理
- 軽微不具合修正

### 月額でやらないこと (追加料金)
- 新ツール作成
- 対象領域外追加
- 外部連携
- 紙資料/OCR

- FAQ新規作成代行
- 契約条件整理代行
- 社内ルール作成
- 緊急対応
- 別部署展開

## 19. 現時点の結論

> 会社別AI保存庫コア + 顧客ごとの業務AIツール
>
> 最初の商品: 保存庫コア + リアルタイム電話/営業回答支援
>
> ただし保存庫コアは汎用基盤・電話支援専用ではない

### 大原則
- 保存庫コアとツール出力を混ぜない
- 顧客資料の正しさは顧客側が決める
- AIが勝手に正式情報を作らない
- 不足/矛盾/最新版不明を検出して提示
- 対象領域内の顧客別条件・例外条件は削らない
- 初期は営業/電話支援が有力だが、保存庫コアは汎用基盤として作る
