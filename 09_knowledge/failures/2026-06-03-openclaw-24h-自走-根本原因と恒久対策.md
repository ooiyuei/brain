# OpenClaw 24時間自走が「途中で止まる」根本原因と恒久対策

> 大井の依頼: 「PCでタスクをぶん回してるけど途中で止まる。永遠にぶん回したい。失敗から学んで環境全体を最強にして」(2026-06-02深夜〜06-03)
> 結論: **止まる主因は "8GB GPU に乗らない qwen3.6:latest(23GB) を loadしようとして Ollama がOOMクラッシュ" + "プロデューサ過多でキュー飽和" + "レーン飢餓"。全て管理者権限なしで恒久修正・実証済み。**

## 環境の制約（これが全ての前提）
- GPU: **RTX 5060 = VRAM 8GB のみ**。14B級以上は乗らない。実用上限は **qwen3:8b(5.2GB)**。
- Claude は**管理者権限なし**。Windowsサービス化/タスクS4U化/Machine環境変数は不可 → 「ログイン中24h常時起動PC」モデルで堅牢化。

## 根本原因（5つ・全て裏取り済み）

### 1. 【最重要】qwen3.6:latest(23GB) の OOM クラッシュ
- 本日 Ollama が **unreachable 27回**。worker のVRAMフィットガードは **direct API 経路しか守らず**、`use_agent` → openclaw agent(ACP gateway) 経路が **qwen3.6:latest をロード → 8GB GPU をOOM → Ollama プロセスごと落ちる → 全worker停止**。
- 発生源: `openclaw.json` の research/newbiz/dev エージェントが `model: qwen3.6:latest`、各エージェントの `models.json` レジストリにも qwen3.6 が `reasoning:true` で登録され openclaw が「最高性能」として自動選択。`dispatch.ps1` も super/high/normal を qwen3.6 にマップ。

### 2. レーン飢餓（実質1レーン直列に退化）
- `worker.ps1` の `Test-LaneMatch` が `use_agent=True` を最優先で必ず heavy 判定 → inbox 全230件が heavy に偏り、**light ワーカーが完全遊休**。2レーン並列のはずが1レーン直列でスループット半減。

### 3. num_ctx 無制限（GPU溢れ→激遅→タイムアウト）
- `worker.ps1` の Ollama 呼び出しに `num_ctx` 指定が無く、qwen3:8b が既定32kコンテキストを確保 → **8GBに乗り切らず63%CPUへ溢れ** → 1ジョブ8分超 → タイムアウト。

### 4. プロデューサ/コンシューマ不均衡（自己DoS）
- inbox が常時 ~230件で飽和、worker は ~8件/時しか捌けない。多数の filler/loop/pipeline が1〜15分毎に投入し続け、**投入 > 排出が常態化**。money_loop は4日以上ノーオペ（"skip: inbox>200"）。大井から見て「止まってる」状態。

### 5. 管理外 any ワーカー & 死んだ gateway
- `run_worker.vbs`(Lane未指定=any) が孤児ワーカーを生み、heavy と同一GPUで同時生成 → VRAM過負荷。
- openclaw gateway daemon(port 18789) が **死亡**（Ollama 27回クラッシュの巻き添えで不正状態）→ `use_agent` が毎回 `GatewayTransportError 1006` → 18-40s 浪費して fallback。

## 適用した恒久対策（管理者不要・全て実機適用済み）

| # | 対策 | ファイル | 効果 |
|---|------|---------|------|
| 1 | **qwen3.6:latest を物理削除** (`ollama rm`) | ollama | OOM経路を物理的に封鎖。23GBディスク解放。再導入はGPU増設後 `ollama pull` |
| 2 | openclaw 4エージェント(main/marketing/newbiz/research) を qwen3:8b に統一、各 models.json から qwen3.6 除去、qwen3:8b の contextWindow 40960→8192 | `~/.openclaw/openclaw.json`, `agents/*/agent/models.json` | agent経路のOOM/CPU溢れ根絶 |
| 3 | **worker の use_agent(openclaw agent)経路を無効化** `$EnableAgentPath=$false` | `worker.ps1` | gateway/ACP依存を断ち、direct Ollama(qwen3:8b)へ一本化。毎回の浪費と43%失敗を解消。GPU増設時に戻せる |
| 4 | `num_ctx=8192` を両Ollama呼び出しに付与 | `worker.ps1` | SIZE 16GB→8.1GB、CPU溢れ 63%→23%(GPU 77%)。タイムアウト解消 |
| 5 | `Test-LaneMatch` をプロンプト長ベースに修正(use_agent→heavy固定を廃止) | `worker.ps1` | light/heavy 両レーン稼働、ドレイン2倍 |
| 6 | dispatch のモデルポリシー super/high/normal → qwen3:8b | `dispatch.ps1` | qwen3.6 をキューに書き込むのを源流で停止 |
| 7 | **一元バックプレッシャ**: inbox>=150 で super以外の新規投入をスキップ | `dispatch.ps1` | 全プロデューサが必ず通る関所。1箇所で flood 停止。キューは常に~150で満たされつつ排出=理想的24h状態 |
| 8 | 膨張ガードを 5-*.json 限定→全prefix年齢順に修正 | `queue_guard.ps1` | 高優先(1-*)滞留時もガードが作動 |
| 9 | 管理外 any ワーカー掃除 + `run_worker.vbs`無効化 | `queue_guard.ps1`, `run_worker.vbs.disabled` | GPU奪い合い停止 |
| 10 | gateway daemon を再起動して復活(port 18789) | OpenClaw Gateway task | (worker は使わないが)VSCode ACP用に復旧 |
| 11 | 文字化け源の BOM 付与(11スクリプト) | `*.ps1` | PS5.1がUTF-8として正読→出力パスのmojibake停止 |
| 12 | daily_brain.ps1 の qwen3.6直叩き地雷を qwen3:8b に | `daily_brain.ps1` | 再有効化時のOOM防止 |
| 13 | 会話ログhookを settings.json に集約(絶対pythonパス) | `~/.claude/settings.json` | 2026-05-29から停止していた会話ログを復旧(次セッションから) |

## 実証結果（2026-06-03 01:27〜)
- テストタスク `dispatch→queue→worker→ollama→出力` 全経路で完走。出力「パイプライン正常。」(model=qwen3:8b・**文字化けゼロ**)。
- queue inbox **230→197**（200割れ・排出開始）。done +8/10分 = **~48件/時(旧8件/時の6倍)**。
- Ollama **unreachable / クラッシュ ゼロ**（修正後）。qwen3:8b が 8192ctx で GPU 77%常駐。

## 残課題（大井判断 or 後日）
- **文字化け不可逆破壊 568本文/89ファイル名**（cp932ラウンドトリップでバイト消失・復元不可）→ 隔離推奨。発生源は一部 bg_*/filler が `powershell -Command` で日本語引数を渡す経路(要 -PromptFile化)。
- **inbox 1336件の未昇格バックログ** → 半自動 triage パイプライン(`inbox_triage.ps1`)新設を提案。
- **openclaw main セッションが sticky qwen3.6** → VSCode ACP使用時は `/reset` 必要。
- 量産 filler の優先度が全 high → 本当に精度の要るタスクが埋もれる。優先度設計の見直し。
- GPU増設(VRAM 16GB+)すれば qwen3.6復帰・agent経路再有効化・num_ctx拡大が可能(各修正にGPU増設時の戻し方をコメント明記済)。

## 第2-3ラウンド（同日午後〜夜・最適化＆価値化）

### 最適化（最強環境）
- **`OLLAMA_NUM_PARALLEL=4`→1**（User env＋ollama再起動）: KVキャッシュを num_ctx×4 確保していたため qwen3:8b がGPUに乗り切らず**CPU溢れ(23%/77%)**していた。1にして**6.0GB/100% GPU化**。ワーカーは実質1ジョブ直列なので並列4は無益だった。「8GBの宿命」と思っていたCPU溢れの真因。`MAX_LOADED_MODELS=1`/`KEEP_ALIVE=30m`/`FLASH_ATTENTION=1`も設定。**env変更後はProcess env更新→ollama app再起動が必須**。
- worker.ps1 options: `repeat_penalty=1.25/repeat_last_n=256/top_p=0.9/temperature=0.2/num_predict=3500`（qwen重複出力抑制・精度優先）。
- worker.ps1 context: 各8000字無制限prepend→**合計4000字上限＋指示を末尾に配置**（Ollamaは超過時に先頭を捨てる→指示が切られていた57k字問題を是正）。
- dispatch.ps1: entity自動注入を `$Prompt` のみ走査（prefixの18 wikilink拾い57k肥大を是正）＋ `prompt-prefix-v3.md`（AI臭禁止表）へ切替。
- 起動時 pause/resume 検証: pause=VRAM 6.0→1.4GB解放、resume=snapshotから38タスク正確復元＋100%GPU精度復帰（地雷AutoReviewは復活させない＝snapshot方式の効果）。

### ゲーム自動pause
- `game_guard.ps1`＋`BrainGameGuard`(2分毎): LoL/Valorant等を検知→.paused+VRAM解放、終了で自動resume。手動pauseと`.paused_by_game`で区別。対象は`scripts/game_processes.txt`。

### 価値化（最重要の構造的気づき）
- **生成は回っていたが価値を生んでいなかった**: auto_review 5/27停止、`_inbox`に**81%が重複コピー**（同一タスク×16）、レビュー/磨き/昇格が全て死。brainは肥大していたが育っていなかった。
- **対策**: biz_filler 10分→45分(重複しない適正レート)、重複1061件アーカイブ、**Opusレビュー&磨き上げワークフロー**で事業ドラフトを**平均35-40点→84点**に昇格（3バッチで34本）。
- **Opusレビューが捕捉した致命的欠陥**（qwen3:8b生出力）: 捏造数字(架空の成果%)、架空連絡先(電話/メール)、価格取り違え(5社目に旧価格→値上げ交渉自滅)、架空の返金約束(契約リスク)、架空の顧客の声(コンテストで誇大=信用毀損)、日付ミス(残18日←entity古値)、全文重複、事業誤認、AI臭。→ **qwen下書き→Opusがレビュー・磨き・昇格の分業を恒久化**。
- entity古日付の是正: `entities/eemus.md` の「残18日」を絶対日付に修正(qwenが相対日付をコピーする問題の源断ち)。

## 教訓（横展開）
1. **ハードの上限(VRAM)を超えるモデルは「設定で速くする」のでなく「使わない/消す」のが最強**。中途半端に残すと必ず誰かがロードしてクラッシュさせる。
2. **ガードは全経路に張る**。direct経路だけ守ってもagent経路が素通しなら無意味。
3. **プロデューサ過多の自走は"動いてる風"の空回り**。関所(dispatch)に1つバックプレッシャを置けば全体を律速できる(DRY)。
4. PS5.1 は **BOM無しUTF-8をcp932誤読**する。日本語リテラルを含む.ps1は必ずBOM付きで保存。
