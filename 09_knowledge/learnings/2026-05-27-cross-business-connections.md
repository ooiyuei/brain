# 主力5事業の隠れたシナジー (2026-05-27 発見)

> **「事業多すぎ・フォーカス散漫」と大井が自己認識する15事業を、brain クロス参照で見ると、実は「共通基盤の異なる出口」として整理できる。**
> 各事業を孤立させて見ると分散だが、共通基盤を抽出して見ると単一プラットフォーム。これが横展開戦略の核。

---

## 1. 共通基盤の発見

**AIpaX (会社別AI保存庫コア) = すべての事業の共通基盤候補**

| 事業 | AIpaX コアの応用 | 顧客 |
|---|---|---|
| AIpaX (B2B) | 中小企業の業務知識保存庫 | 経営者 |
| AIpa Web | 採用フォーム→AI構成→LP生成 (= 採用情報の保存庫+生成) | 中小企業の人事/経営者 |
| AIpaX school | 教育コンテンツの個別保存庫+AI採点 (= 学習履歴の保存庫) | 小中高生・親 |
| EEMUS | 高校生×地元企業のマッチングDB+AI推薦 (= 企業情報の保存庫+推薦) | 高校生・静岡企業 |
| Testall | 受験生の学習履歴保存+45分プラン生成 (= 学習プロファイルの保存庫) | 受験生 |

**つまり**: 各事業は「AIpaX コア (保存庫+生成エンジン)」の異なる適用領域。バックエンドは1つに統合可能。

---

## 2. 横展開の戦略的含意

### 含意A: AIpaX コアを「内部マルチテナント基盤」化すれば横展開が指数関数的に加速

- 現状: 各事業を独立に作ってる (15事業 = 15個のコード/DB/UI)
- 提案: AIpaX コアを Supabase ベースの **マルチテナントSaaS基盤** に統一し、各事業はその上の「テナント別UI」として実装
- 効果: 新規事業立ち上げが「DBスキーマ + フロントUI」だけで済む (バックエンド開発不要)
- 既に[[wiki/meta/shincoder-actions-for-ooi-2026-05-25]] G1 で Turso 検討あり。Turso なら multi-tenant 1000枠/100枠で安価

### 含意B: 顧客クロスセル

- AIpaX 顧客 (B2B経営者) → 子供がいる場合、AIpaX school へクロスセル可能
- AIpa Web 顧客 (採用課題) → EEMUS (高校生インターン送り込み) と接続可能
- AIpaX school 受講生 → 親が経営者なら AIpaX へクロスセル可能
- AIpa Web 顧客 (静岡企業) → AIpaX (業務AI導入) へアップセル可能

**仮説**: 1顧客あたりの LTV を 65万 → 200万 (3倍) に伸ばす可能性あり (クロスセル)

### 含意C: ブランド統一の威力

- 現状: AIpaX / AIpa Web / AIpaX school と「AIpa」系統で命名統一済み
- だが顧客から見ると別物の3サービス
- ブランドストーリーを「AIpa = 中小企業・若者・地方の AI 民主化プラットフォーム」で統一し、各事業をその下位ブランドとして展開すれば、認知獲得コストが下がる

## 3. 大井の認知特性との接続

[[09_knowledge/learnings/2026-05-27-ooi-integrated-portrait]] によれば:

- 大井の強み: 「アイデア生成 (CPU/GPU) >> 実装 (RAM)」
- 弱み: 15事業並走で「実装処理が追いつかない」「フォーカス散漫」

**横展開戦略の認知的合理性**:
- 15事業を孤立して見ると RAM が足りない (= 焦り・自己否定の原因)
- 共通基盤で見ると 1基盤 × 15 UI で RAM 負担が 1/15
- 大井の CPU/GPU 上位特性が活きる構造 (基盤判断は1回・適用先選定は瞬時)

## 5. 警告とリスク

| リスク | 対処 |
|---|---|
| 共通基盤の早期最適化 = 過剰設計の罠 | まず AIpaX + AIpa Web の2テナントだけ実装、3本目から判断 |
| 大井の「フォーカス散漫」が共通基盤検討に流れ込む | 「実装の主体は架橋者=相方」、大井は判断のみ |
| 顧客側ニーズの差異を無視するとどの顧客にも刺さらない | 顧客との対話は分離維持。基盤は内部のみ |
| AIpaX の B2B 顧客と AIpaX school の親顧客では「AI」の意味が違う | ブランド戦略は分離、技術基盤のみ統一 |
