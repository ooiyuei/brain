# CPA で需要を判定するルール

> **CPA = Cost Per Acquisition = 1人の顧客 (ウェイトリスト/契約/購入) を獲得するためにかかるコスト**
> シンコーダー流: **「LP作って広告1万円→CPA測定」が最も再現性高い需要検証**

---

## CPA の計算

```
CPA (円) = 広告費 (円) ÷ 獲得数 (人)
```

### 例
- 広告費 10,000円 / ウェイトリスト 100人 → CPA 100円 ✅
- 広告費 10,000円 / ウェイトリスト 62人 → CPA 161円 ✅ (大井のサブスクボックス LP相当)
- 広告費 10,000円 / ウェイトリスト 10人 → CPA 1,000円 ❌
- 広告費 10,000円 / ウェイトリスト 0人 → 即ピボット ❌

---

## 判定基準 (シンコーダー基準)

| CPA | 判定 | 行動 |
|---|---|---|
| ≦ 100円 | 🔥 大バズ確実 | 即開発GO + マーケ強化計画 |
| 100-300円 | ✅ **確実需要あり** | 開発GO |
| 300-500円 | 🟡 需要可能性あり | 判断保留 / 再検証 / 進む |
| 500-1000円 | ⚠️ 需要薄い | ターゲット見直し or ピボット |
| 1000円+ | ❌ ピボット推奨 | アイデア/UI/コピー変えて再検証 |

注: この閾値は「ウェイトリスト集めCPA」の話。実購入CPAなら 1-2桁高い (LTV依存)。

---

## 推奨検証セット (1事業 1セット 2万円)

### A. X (Twitter) 広告 1万円
- 期待: 30万インプレッション (シンコーダー実績)
- ターゲット: 関連興味カテゴリ + 地域
- 期間: 7日間
- フォロワー0でも実行可能

### B. メタ広告 (Facebook/Instagram) 1万円
- 期待: より購買意欲の高い層にリーチ
- ターゲット: より細かい属性絞り込み可
- X と組み合わせで多角度の検証

### LP には必ず以下を配置
1. **Form Spree のウェイトリスト** (メアド収集)
2. CTA: 「興味あったらメアド入れて。先行案内します」
3. シンプルで1秒で理解できる FV (ファーストビュー)

## 大井の実例 (2026年時点)

### サブスクボックス (シンコーダー実例 ※大井ではない)
- 広告 10,000円 → ウェイトリスト 62人 → CPA 161円 ✅
- → 開発GO → 月40万円売上達成

### AIpa Web (大井 既存事業)
- 5/26 着弾の LP案 + ウェイトリスト機能の Form Spree あり
- **次のアクション**: 6/2 までに X広告1万円投化 → CPA測定
- 結果次第で 6/3 にGO/PIVOT判定

### AIpaX (大井 既存事業)
- 既に 4社契約済 (= 検証フェーズ突破)
- **5社目用に新LP作成 → X広告1万円投化 → CPA測定** (5/31 期限)
- shincoder-actions A1 と一致

## CPA測定の失敗パターン (回避)

### ❌ 失敗1: ターゲット広すぎ
- 「日本人全員」だと CPA 跳ね上がる
- 解決: 業界・地域・年齢・興味カテゴリで絞る

### ❌ 失敗2: LP のFVが分かりにくい
- ユーザーは1秒で離脱
- 解決: GPT image 2.0 で画像生成 + Claude Design で構成

### ❌ 失敗3: ウェイトリストCTAが弱い
- 「メアド入れて」だけだと登録動機が薄い
- 解決: 「先行案内 + モニター3社限定 + 通常価格より50%オフ」等の特典明示

### ❌ 失敗4: 広告を即停止
- 7日間は回す (アルゴリズム最適化に時間がかかる)
- 結果は7日後に評価

## Claude-CEO の運用ルール

- 大井が新事業を立ち上げる時 → 必ず Step 1-4 を実行 (CPA測定なしの開発禁止)
- 既存事業の停滞時 → CPA再測定で需要再確認
- 結果を `01_ceo/decisions/cpa-log.md` に時系列で記録 (将来の判断材料)
